ai_news_scraping
운영 셋업 가이드

대표님께 — ralph 가 코드를 다 만들었습니다. 이제 외부 API 키 발급 + 실 발송 1회 + GitHub Actions cron 가동까지가 운영자(대표님) 영역입니다. 이 문서를 위에서 아래로 순서대로 따라하시면 약 40~60분 안에 매일 아침 자동 발송이 가동됩니다.

1개요 & 아키텍처

이 서비스는 매일 아침 영어권 AI 매체를 자동 수집·요약해 한국어로 메일 전송합니다 (CLAUDE.md §1 비전).

┌───────────────────────────────────────────────────────────┐ │ GitHub Actions cron (매일 23:40 UTC = 08:40 KST) │ └──────────────────────────────┬────────────────────────────┘ ▼ ┌───────────────────────────────────────────────────────────┐ │ python -m ai_news_scraping.cli run │ └──────────────────────────────┬────────────────────────────┘ ▼ 1) Google Custom Search (키워드 5 × 매체 10 → 1 호출/kw) 2) trafilatura 본문 추출 (실패 매체는 자동 skip) 3) Supabase articles 저장 (url 유일 키로 dedup) 4) Gemini 통합 요약 (영문 → 한국어 트렌드 정리) 5) Gmail SMTP BCC 발송 (구독자 ~10명) Admin (FastAPI 단일 HTML) ───────────────────────── · 자동 스크래핑 ON/OFF 토글 · 구독자 명단 추가/제거
📌 비용: 모든 외부 서비스가 무료 tier 안에서 운영됩니다 (CLAUDE.md §6 / §7). Google CSE 일 100회, Supabase 500MB, Gemini Flash 무료 tier, GitHub Actions 무료.

2-1. Brave Search API 필수

💡 왜 Brave?: 이전 버전은 Google Custom Search 였으나 신규 Cloud 프로젝트에서 PERMISSION_DENIED 가 빈번하고 발급 단계가 복잡해 (Cloud project + Library enable + Programmable Search Engine + cx) 운영 신뢰도가 낮았습니다. Brave Search 는 월 2,000 호출 무료 (우리 케이스 일 5 × 30 = 150 호출의 13배 여유) 이고 발급도 3분 안에 끝납니다.

발급 절차

  1. https://api-dashboard.search.brave.com/ 접속 → 우상단 Sign up (Google / GitHub / 이메일 로그인).
  2. 좌측 메뉴 Subscriptions → "Free" plan 의 "Subscribe" 클릭.
  3. 카드 정보 입력
    ⚠️ 카드 등록: Brave Free 가 카드 정보를 요구하긴 하지만 quota 초과 시 자동 차단이라 청구 0. 월 2,000 안에서 운영하는 한 비용 발생 X. 우리 호출량 (월 150) 의 13배 여유.
  4. 좌측 메뉴 API Keys → "Add API Key" 클릭.
  5. 이름 임의 (예: ai-news-scraping) → Add.
  6. 발급된 토큰을 메모장에 복사 → 이것이 BRAVE_SEARCH_API_KEY.
📌 매체 화이트리스트: Google CSE 와 달리 Brave 는 별도 "검색엔진" 객체를 만들 필요가 없습니다. 매체 화이트리스트는 우리 코드 (domains/ai_news/sources.yaml) 가 자동으로 site:(d1 OR d2 ...) 형식으로 쿼리에 주입.
✅ 결과: BRAVE_SEARCH_API_KEY 한 값만 메모장에 보관. Google CSE 처럼 cx 별도 발급 안 함.

2-2. Gemini API 필수

  1. https://aistudio.google.com/app/apikey 접속.
  2. Google 계정 로그인 후 "Create API key" 클릭.
  3. (선택) "Create API key in new project" 또는 위에서 만든 프로젝트 선택.
  4. 발급된 키 복사 → GEMINI_API_KEY.
📌 모델: 기본은 gemini-2.5-flash. 무료 tier 로 충분합니다. 모델을 바꾸고 싶다면 .env 의 GEMINI_MODEL 값을 수정.

2-3. Gmail 앱 비밀번호 필수

⚠️ 2FA 활성화 필수: 앱 비밀번호는 2단계 인증이 켜진 계정에서만 발급 가능합니다. https://myaccount.google.com/security → "2단계 인증" 켜기.
  1. 발송에 사용할 Gmail 계정으로 로그인.
  2. https://myaccount.google.com/apppasswords 접속.
  3. 앱 이름 입력 (예: ai-news-scraping) → "Create" 클릭.
  4. 16자리 비밀번호가 표시됨. 공백 제거해서 16글자 문자열로 복사 → GMAIL_APP_PASSWORD.
  5. 해당 Gmail 주소 → GMAIL_USER (예: your-account@gmail.com).
🚨 보안: 앱 비밀번호는 발급 후 한 번만 표시됩니다. 분실 시 재발급 후 .env 갱신. 절대 git 에 커밋하지 마십시오 (.gitignore 에 .env 포함됨).

2-4. Supabase 프로젝트 필수

  1. https://supabase.com/ 접속 → "Start your project" → GitHub/Google 로그인.
  2. 대시보드 우상단 "New project" 클릭.
  3. 입력값:
    • Name: ai-news-scraping
    • Database password: 강력한 임의 값 (어딘가 안전한 곳에 보관)
    • Region: Northeast Asia (Seoul) 또는 Northeast Asia (Tokyo)
    • Pricing: Free
  4. 프로젝트 생성 완료 후 (1~2분 소요), 좌측 메뉴 Settings → API.
  5. 다음 두 값을 복사:
    • Project URL (예: https://abcdefghij.supabase.co) → SUPABASE_URL
    • service_role key (Project API keys 섹션, "Reveal" 클릭) → SUPABASE_SERVICE_ROLE_KEY
🚨 매우 중요: service_role 키는 RLS 우회 권한을 가집니다. 절대 클라이언트(브라우저)에 노출 금지. 이 서비스는 서버사이드에서만 사용합니다.

2-5. Admin token 필수

Admin 페이지 (스크래핑 ON/OFF + 구독자 관리) 인증용 임의 토큰. 로컬에서 생성:

openssl rand -hex 32

출력된 64자리 hex 문자열을 ADMIN_TOKEN 으로 사용.

✅ 2단계 완료 체크리스트

  • BRAVE_SEARCH_API_KEY
  • GEMINI_API_KEY
  • GMAIL_USER (보낼 Gmail 주소)
  • GMAIL_APP_PASSWORD (16자리)
  • SUPABASE_URL
  • SUPABASE_SERVICE_ROLE_KEY
  • ADMIN_TOKEN (64자리 hex)

총 7개 값이 메모장에 있으면 다음 단계로.

3Supabase 마이그레이션 적용

7개의 테이블 (articles / subscribers / runs / scrape_enabled / search_keywords / search_sources / search_settings) 을 ai_news schema 안에 만듭니다. 마이그레이션은 2개 (0001 = 핵심 4개, 0002 = admin 검색 조건 3개).

🔒 schema 격리: 같은 Supabase 프로젝트에 다른 서비스가 들어와도 격리되도록 별도 schema (ai_news) 안에 둡니다 (CLAUDE.md §6 원칙). 마이그레이션이 create schema if not exists ai_news; 부터 시작해 권한 부여까지 자동 처리합니다.

방법 A — Dashboard SQL Editor (권장, 가장 단순)

  1. Supabase 프로젝트 대시보드 → 좌측 메뉴 SQL Editor.
  2. 우측 상단 "+ New query" 클릭.
  3. 0001 적용: 로컬 supabase/migrations/0001_initial_schema.sql 전체를 SQL 에디터에 붙여넣기 → Run.
  4. 0002 적용: supabase/migrations/0002_search_admin.sql 동일 절차로 적용 (search_keywords / search_sources / search_settings).
  5. 우측 하단 "Run" 버튼 클릭 (또는 ⌘ + Enter).
  6. "Success. No rows returned" 메시지 확인.
  7. 좌측 메뉴 Table Editor 클릭 → 좌상단 schema 드롭다운을 public 에서 ai_news 로 변경 → 다음 7개 테이블이 보이는지 확인:
    • articles · subscribers · runs · scrape_enabled (0001)
    • search_keywords · search_sources · search_settings (0002, admin 운영용)
    ⚠️ schema 드롭다운: 처음에는 public 만 보이므로 "테이블이 없다" 고 당황할 수 있습니다. 드롭다운을 ai_news 로 바꾸시면 보입니다.
  8. Exposed schemas 설정 (필수): 좌측 메뉴 Settings → API → "Exposed schemas" 필드에 ai_news 를 추가 (기존 public, graphql_public 옆에 콤마로 이어붙임) → Save.
    🚨 이 단계 안 하면: supabase-py 가 PostgREST 통해 ai_news schema 에 접근 못 합니다 (40x 에러). 반드시 추가.

방법 B — Supabase CLI (개발자용)

# 설치 (1회)
brew install supabase/tap/supabase

# 프로젝트 연결
supabase login
supabase link --project-ref <your-project-ref>

# 마이그레이션 적용
supabase db push

project-ref 는 Supabase URL 의 서브도메인 (예: abcdefghij).

✅ 3단계 완료 체크리스트

  • Table Editor 에 4개 테이블 보임
  • scrape_enabled 에 id=1, enabled=true 1행 존재

4로컬 셋업

로컬에서 한 번 dry-run + 실 발송이 동작해야 GitHub Actions 도 동일하게 동작합니다.

4-1. 사전 도구 설치 (1회)

Python 패키지 매니저 uv 가 필요합니다 (Python 가상환경 + 의존성 관리).

# macOS
brew install uv

# 확인
uv --version
# 출력 예: uv 0.9.29

4-2. 저장소 준비

cd ~/jinsup_ralph/ai_news_scraping
# 이미 있으면 위 디렉토리로 이동만

uv sync --dev

처음 실행 시 Python 3.12 자동 다운로드 + 패키지 90여개 설치 (1~3분).

4-4. .env 파일 작성

cp .env.example .env

편집기로 .env 를 열고 각 값을 2단계에서 발급한 실제 값으로 채웁니다:

# Supabase

# 운영 플래그

4-5. 검증 (lint + typecheck + tests)

uv run ruff check .
uv run mypy
uv run pytest

모두 exit 0 이고 pytest 가 130+ 개 통과해야 합니다.

✅ 4단계 완료 체크리스트

  • uv --version 동작
  • .env 8개 값 모두 채워짐
  • pytest 130+ 통과

5로컬 dry-run (실 발송 없이 흐름 검증)

검색·본문 추출·Gemini 요약까지만 수행하고 메일 발송은 skip 합니다. Supabase 의 articles 테이블에는 그날 수집된 기사가 실제로 저장됩니다.

uv run python -m ai_news_scraping.cli run --dry-run

예상 로그 (정상)

# 약 1~3분 소요
2026-05-23 09:30:00 INFO ai_news_scraping.pipeline: ...
2026-05-23 09:32:00 INFO ai_news_scraping.cli: run_id=... status=skipped search=N extracted=M articles=10 accepted=0 refused=0

status=skipped + skipped_reason=dry_run 이면 정상. articles=N 이 10~20 안이면 잘 작동.

Supabase 확인

  1. Supabase 대시보드 → Table Editor → articles.
  2. 오늘 날짜로 row 가 추가됐는지 확인. body_text 칼럼에 영문 기사 본문이 보여야 함.
  3. runs 테이블은 현재 자동 기록되지 않음 (v2 개선 후보) — 이번 검증에서는 무시.
🚨 문제 발생 시:
  • articles=0 → 검색이 0건. 키워드/매체 화이트리스트 점검 또는 dateRestrict=d1 이 너무 좁을 수 있음. FAQ 참조.
  • ValidationError: ...field required → .env 가 비어있음.
  • 401 / 403 → API 키 오타 또는 권한 미부여.

✅ 5단계 완료 체크리스트

  • --dry-run 명령이 exit 0
  • articles 테이블에 오늘 row 가 다수 보임
  • body_text 가 비어있지 않음

6본인 메일로 실 발송 1회

구독자 명단에 대표님 본인 1명만 등록한 상태에서 1회 실제 발송 → 메일 도착 확인.

6-1. 본인을 구독자로 등록

Supabase Dashboard 에서 직접 입력하는 게 가장 빠릅니다:

  1. Supabase → Table Editor → subscribers → 우상단 "+ Insert row".
  2. 입력:
    • email: 본인메일@gmail.com (혹은 다른 도메인)
    • active: true (체크박스)
  3. Save.

또는 admin 페이지로 등록 (9단계 참고).

6-2. dry-run 빼고 실행

uv run python -m ai_news_scraping.cli run

1~3분 후 로그에 status=success accepted=1 refused=0 이 보여야 함.

6-3. 받은 메일 확인

  1. 등록한 메일함을 엽니다 (스팸함도 확인).
  2. 제목: 오늘의 AI 트렌드 (2026-MM-DD) 형식.
  3. 본문이 한국어 마크다운 (HTML 렌더링) 으로 보임 — 트렌드 요약 + 출처 링크.
⚠️ 메일이 안 오면:
  • 스팸함 확인 (Gmail 이 자기 자신이 보낸 메일을 스팸으로 분류할 수 있음).
  • 로그에 refused 가 있는지 확인.
  • SMTP 인증 실패 시: 앱 비밀번호 재발급 + .env 갱신.

✅ 6단계 완료 체크리스트

  • 로그 status=success accepted=1 refused=0
  • 본인 메일함에 디지스트 도착
  • 본문 한국어, 트렌드 단위로 묶임, 출처 링크 포함

7GitHub Actions 배포 (매일 자동 실행)

로컬에서 검증이 끝났으니 이제 GitHub Actions cron 으로 매일 자동 실행되게 합니다.

7-1. GitHub 저장소 준비

아직 GitHub 저장소가 없다면:

# GitHub 에 새 private 저장소 생성 (web UI) — 이름 예: ai_news_scraping

# 로컬에서 push
cd ~/jinsup_ralph/ai_news_scraping
git remote add origin https://github.com/your-username/ai_news_scraping.git
git branch -M main
git push -u origin main
⚠️ private 권장: public 저장소에 올리면 코드는 보안에 문제 없으나, GitHub Actions 무료 분(分) 정책이 약간 다릅니다. private 가 무난.

7-2. Secrets 등록 (8개)

  1. GitHub 저장소 페이지 → Settings 탭.
  2. 좌측 메뉴 → Secrets and variables → Actions.
  3. 우측 "New repository secret" 버튼 클릭.
  4. 다음 8개를 하나씩 추가 (이름은 그대로, value 는 .env 의 실제 값):
    BRAVE_SEARCH_API_KEY.env 와 동일
    GEMINI_API_KEY.env 와 동일
    GMAIL_USER.env 와 동일
    GMAIL_APP_PASSWORD.env 와 동일 (공백 없는 16자리)
    SUPABASE_URL.env 와 동일
    SUPABASE_SERVICE_ROLE_KEY.env 와 동일
    ADMIN_TOKEN.env 와 동일
📌 선택사항: Gemini 모델을 바꾸려면 같은 화면의 Variables 탭에서 GEMINI_MODEL 추가. 기본은 gemini-2.5-flash.

7-3. 수동 트리거로 1회 검증 (dry-run)

  1. 저장소 페이지 → Actions 탭.
  2. 좌측에서 "Daily AI News Digest" 워크플로우 선택.
  3. 우측 상단 "Run workflow" 드롭다운.
  4. dry_run 체크박스 ON → "Run workflow" 클릭.
  5. 1~5분 후 워크플로우가 ✅ 녹색으로 완료되는지 확인.
  6. 실패 시 로그 클릭해서 에러 메시지 확인 — secrets 누락이 가장 흔한 원인.

7-4. 수동 트리거로 실 발송 1회

  1. 같은 메뉴에서 "Run workflow" → 이번엔 dry_run OFF.
  2. 완료 후 등록된 구독자 메일함에 디지스트 도착 확인.

✅ 7단계 완료 체크리스트

  • 8개 secrets 등록
  • dry_run=true 트리거 ✅ 성공
  • dry_run=false 트리거 ✅ 성공 + 메일 도착

8cron 자동 실행 확인 (다음날 아침)

이제 별도 조작 없이 매일 08:40 KST (= 23:40 UTC) 에 자동 실행됩니다.

스케줄 검증

  1. 오늘 밤 잠들기 전 노트북·서버 모두 꺼져 있어도 무관 (GitHub Actions 가 클라우드에서 실행).
  2. 다음 날 09:00 ~ 09:30 사이 메일함 확인 → 디지스트 도착 여부.
  3. Actions 탭에서 워크플로우 실행 로그 확인 (스케줄 트리거된 run).
⚠️ GitHub Actions cron 지연: 플랫폼 부하가 높을 때 정시에서 최대 ~15분 지연될 수 있습니다. CLAUDE.md §4 의 "08:40 ± 30분" 허용 윈도우 안이라 무관.

  1. 브라우저: http://127.0.0.1:8000.
  2. HTTP Basic 인증 팝업:

    • Password: ADMIN_TOKEN 값.
  3. 화면:

일시 중단

휴가 등으로 며칠 안 받고 싶으면 Admin 페이지 → "토글" 버튼 → OFF. 다음번 cron 이 자동으로 skip. 재개는 다시 토글 ON.

📌 영구 운영 호스팅: Admin 페이지를 항상 켜놓고 싶으면 Vercel / Fly.io 같은 무료 호스팅에 배포 가능. 그러나 ~10명 운영이면 필요할 때 로컬에서 실행해도 충분합니다.

10문제 해결 FAQ

Q1. --dry-run 에서 articles=0 이 나옵니다.

원인 후보:

  • dateRestrict=d1 (최근 1일) 이 너무 좁아서. 코드의 cli.py 호출에서 PipelineParams.date_restrict 를 "d7" (7일) 또는 "w1" (1주) 로 임시 조정.
  • 매체 화이트리스트의 도메인이 검색 결과에 안 나타남. domains/ai_news/sources.yaml 점검.
  • 키워드가 너무 niche. domains/ai_news/keywords.yaml 의 키워드를 일반적으로.

Q2. SMTP 인증 실패

Gmail 앱 비밀번호는 발급된 직후 한 번만 표시됩니다. 16자리에서 공백을 제거해서 .env 와 GitHub Secrets 에 저장했는지 확인.

Q3. Supabase 일시중지 알림이 옵니다.

무료 tier 는 7일간 무활동 시 프로젝트 일시중지됩니다. 매일 cron 이 doing 도는 한 활성 유지됩니다. 휴가로 OFF 토글한 동안에는 dashboard 에 한 번 접속하거나 수동 쿼리 1회 실행이면 활성 유지.

Q4. Gemini 응답이 영어로 나옵니다.

모델이 prompt 의 "한국어로 작성" 지시를 무시한 케이스. 모델을 gemini-2.5-pro 로 일시 변경 후 테스트. .env 또는 GitHub Variables 의 GEMINI_MODEL 갱신.

Q5. 매일 발송 시각을 바꾸고 싶습니다.

.github/workflows/daily-digest.yml 의 cron: "40 23 * * *" 수정. UTC 기준이라 KST 에서 9시간 빼서 입력. 예: 06:30 KST = 21:30 UTC 전날 = 30 21 * * *.

Q6. 다른 도메인 (예: 헬스케어) 에도 쓰고 싶습니다.

이 코드베이스는 도메인 분리 가능하게 설계됐습니다 (CLAUDE.md §3.1).

mkdir -p domains/health_news
# keywords.yaml + sources.yaml 만들고

# GitHub Actions workflow 복사 후 --domain 인자 추가
uv run python -m ai_news_scraping.cli run --domain health_news

Q7. 구독자가 늘어나면 메일 발송이 느려집니까?

BCC 일괄 발송이라 1회 SMTP 호출. ~10명 → ~100명 까지는 큰 차이 없음. Gmail SMTP 의 일일 발송 한도 (~500건/일) 안.

Q8. 본문 추출이 실패하는 매체가 있습니다.

trafilatura 가 못 뜯어내는 매체 (paywall, JS 렌더링 등) 는 자동 skip 됩니다. 로그에 extract skipped: <url> 가 반복되면 domains/ai_news/sources.yaml 에서 해당 매체 제거.
