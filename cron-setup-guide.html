SETUP GUIDE · 2026-05-24

cron 자동 발송
+ admin 시각 설정

GitHub Actions cron 을 워크플로 파일 hardcode 시각 (23:40 UTC) 에서 해방시키고 admin Settings 에서 발송 시각 (HH:MM KST) 을 직접 설정 가능하게 만든 피처입니다. 이 가이드는 대표님이 손수 해야 하는 3단계 (secrets 등록 → 마이그레이션 → 첫 트리거) 와 운영 중 시각 변경 방법을 담고 있습니다.

● 추가 비용 0원 ⏱ 대표님 손 ~15분 📦 GitHub Actions free tier 36%

01개요 — 무엇이 바뀌는가

한 줄: cron 이 매 5분 깨어나고, admin 의 시각 설정과 ±2분 매칭 시점만 진짜 발송한다.

기존 (hardcode)

.github/workflows/daily-digest.yml 의 cron: "40 23 * * *" 가 매일 UTC 23:40 = KST 08:40 단 1회 trigger. 발송 시각 변경 = workflow 파일 수정 + git commit + push.

신규 (admin 제어)

cron 이 '*/5 23,0 * * *' 로 변경 — KST 08:00 ~ 09:55 사이 매 5분 깨어남 (일 22회). runner 가 admin DB 의 send_hour:send_minute 읽어 현재 KST 와 ±2분 윈도우 매칭 시점에만 파이프라인 진행. 그 외 trigger 는 즉시 skip (DB row 추가 X).

왜 이 설계? workflow 파일을 GitHub API 로 동적 수정하려면 PAT 권한 + 매번 commit 발생 — 무거움. 매 5분 sweep 은 free tier 의 36% (월 720분) 만 사용. admin 변경은 즉시 반영 (다음 cron trigger 부터).

02동작 흐름

cron trigger 부터 메일 발송까지 5 게이트.

┌─────────────────────────────────────────────────────────────────┐ │ GitHub Actions cron '*/5 23,0 * * *' UTC │ │ KST 08:00, 08:05, 08:10, ..., 09:55 (일 22회 trigger) │ └────────────────────────────────┬────────────────────────────────┘ ▼ ┌──────────────────────────┐ │ cli.run_command() │ └──────────┬───────────────┘ ▼ ┌──────────────────────────────────┐ │ 게이트 ① scrape_enabled? │ │ OFF → return │ └──────────┬───────────────────────┘ ▼ ┌──────────────────────────────────┐ │ 게이트 ② subscribers 0? │ │ yes → return │ └──────────┬───────────────────────┘ ▼ ┌──────────────────────────────────┐ │ 게이트 ③ 윈도우 안? ⭐NEW │ │ |now − target| > 5분 → return│ │ (target = admin send_hour:m) │ └──────────┬───────────────────────┘ ▼ ┌──────────────────────────────────┐ │ 게이트 ④ 오늘 success? ⭐NEW│ │ 이미 있음 → return │ └──────────┬───────────────────────┘ ▼ ┌──────────────────────────────────┐ │ ▶ pipeline.run() │ │ 검색→추출→DB→Gemini→Gmail BCC │ └──────────────────────────────────┘ 강제발송 (admin 버튼) = force=True → ③④ 무시 로컬 dry-run = dry_run=True → ③④ 무시 (테스트)

03GitHub Secrets 8개 등록

위치: GitHub repo → Settings → Secrets and variables → Actions → New repository secret.

이미 .env 에 있는 값을 그대로 복붙하시면 됩니다 (양쪽 동일하게 유지).

BRAVE_SEARCH_API_KEY
Brave Search 검색 API 키.
FROM .env
GEMINI_API_KEY
Google AI Studio 발급 키.
FROM .env
GMAIL_USER
발신용 Gmail 주소 (대표님 메일).
FROM .env
GMAIL_APP_PASSWORD
Gmail 앱 비밀번호 16자리 (공백 제거).
FROM .env
SUPABASE_URL
Supabase 프로젝트 URL.
FROM .env
SUPABASE_SERVICE_ROLE_KEY
RLS 우회용 service_role JWT.
FROM .env
ADMIN_TOKEN
admin 페이지 Basic Auth 토큰 (cron 자체엔 불필요하지만 secret 자리 일관성 유지).
FROM .env
SUPABASE_SCHEMA
"ai_news" 고정. workflow vars 또는 secret 어디든. 코드 default 도 "ai_news" 라 미등록도 OK.
OPTIONAL
🔒 보안 SUPABASE_SERVICE_ROLE_KEY 는 RLS 우회 권한이라 절대 클라이언트 / 공개 저장소 노출 금지. .env 는 이미 .gitignore 에 포함되어 있고 GitHub Secrets 는 commit 안 됨.
  1. repo 진입 대표님 GitHub repo 의 Settings 탭 클릭.
  2. Actions secrets 탐색 좌측 메뉴 Secrets and variables → Actions 클릭.
  3. Repository secrets 추가 New repository secret 버튼 → Name + Value 입력 → Add secret 반복 (위 8개). Value 는 한 번 등록하면 다시 못 보니까 .env 와 정확히 동일하게 복붙.
  4. 등록 확인 등록 후 화면에 secret 이름만 노출되고 값은 *** 마스킹. 8개 모두 보이면 완료.

04Supabase 마이그레이션 0004

search_settings 테이블에 send_hour, send_minute 컬럼 추가.

  1. Supabase Dashboard 접속 프로젝트 → SQL Editor 탭.
  2. 마이그레이션 SQL 복붙 다음 SQL 그대로 paste + Run: 
    alter table ai_news.search_settings
    add column if not exists send_hour smallint not null default 8
      check (send_hour between 0 and 23),
    add column if not exists send_minute smallint not null default 40
      check (send_minute between 0 and 59);
    (원본 파일: supabase/migrations/0004_send_schedule.sql)
  3. 적용 확인 Table Editor → schema: ai_news → search_settings 선택 → send_hour=8, send_minute=40 컬럼이 생긴 row 1개 보이면 OK.
⚠️ 멱등성 if not exists 가 있어 두 번 실행해도 안전. 기존 데이터 손실 0.

05수동 트리거 1회 — 정상 동작 검증

cron 자동 실행을 기다리기 전에 수동으로 한 번 돌려 secrets 등록 + 마이그레이션이 잘 됐는지 확인.

  1. Actions 탭 진입 GitHub repo → Actions 탭 → 좌측 Daily AI News Digest workflow 클릭.
  2. Run workflow 클릭 우측 상단 Run workflow 버튼 → branch=main 확인 → Run workflow. (옵션 dry_run 체크하면 메일 발송 없이 흐름만 검증.)
  3. 로그 관찰 실행 중인 run 클릭 → digest job → 로그 확인. 다음 패턴이 보이면 정상: 
    # 시각 매칭 시점이 아니면 즉시 skip — 정상 동작
send-schedule gate: outside window (target=08:40 KST, now=14:23 KST), skipping

# 매칭 시점이고 메일 발송 진행
run_id=... status=success search=24 extracted=18 articles=12 accepted=10
  4. 메일 수신 확인 본인 Gmail 받은편지함에 한국어 요약 메일이 도착했으면 완료. 매칭 시점 외 trigger 라서 outside window skip 만 나면 정상 (cron 매 5분 sweep 중 1번만 발송).
💡 첫 검증 팁 바로 결과 보려면 admin 페이지에서 ▶ 강제발송 클릭 (cron 무관 즉시 진행). 또는 admin Settings 의 send_hour:send_minute 을 현재 KST 시각 근처로 임시 변경 → 다음 10분 trigger 에서 발송 → 다시 8:40 으로 복원.

06admin 에서 발송 시각 변경

발송 시각은 admin Settings 탭에서 즉시 변경. workflow 파일 / git push 불필요.

UI 위치

make admin → http://127.0.0.1:6661/#settings 로 진입 (또는 운영 서버 admin URL).

입력 폼

Settings 탭의 마지막 필드 "발송 시각 (KST)" 칸 2개 (hour : minute).

  • hour: 0 ~ 23 (24시간 표기). 권장: KST 08~09 (cron 윈도우 안).
  • minute: 0 ~ 59.
  • 밖에 둘 수도 있지만 cron sweep 윈도우 (08:00~09:55) 안에 없으면 발송 안 됨.

저장 효과

저장 버튼 클릭 → 다음 cron trigger 부터 즉시 반영. 잘못된 값 (24 / 60 / 음수 등) 은 400 응답 + 폼 에러로 reject.

예시 — 09:15 KST 로 변경

  1. Settings 탭 → 발송 시각 hour=9, minute=15 입력.
  2. 저장 클릭. 페이지 자동 새로고침.
  3. Overview 탭 첫 행 "발송 시각: 09:15 KST" 표시 확인.
  4. cron 09:15 trigger 가 윈도우 (09:13~09:17, ±2분) 의 정확 매칭이라 통과 → 메일 발송.
⚠️ 윈도우 범위 밖 cron 은 KST 08:00 ~ 09:55 만 sweep. 예를 들어 send_hour=14 로 저장하면 cron 이 14:00 에 깨지 않아 발송 X. 14시 발송이 필요하면 .github/workflows/daily-digest.yml 의 cron 값을 추가 변경 (코드 변경 필요).

07cron 자동 동작 확인 (다음날)

수동 트리거 성공 후 다음날 자동 발송이 정상 동작하는지 확인.

  1. 다음날 08:40 KST 전후 메일 확인 기본 설정 (8:40) 기준 KST 08:35~08:45 사이 메일 도착. 받은편지함의 한국어 요약 메일.
  2. admin History 탭 확인 admin → History → 최근 run 1개가 status=success, article_count=12~20, digest preview 표시되면 OK.
  3. GitHub Actions 탭에서 skip 패턴 확인 Actions → Daily AI News Digest 의 최근 run 12개 (KST 08:00 ~ 09:55) 중 1개만 ✅ success, 나머지 11개는 ✅ success 이지만 로그에 outside window 또는 already sent today skip 메시지. (success 처럼 보이는 이유: run_command 가 exit 0 으로 깨끗하게 종료. 실제 발송은 1회만.)

08하루 타임라인 (참고)

기본 설정 (send_hour=8, send_minute=40) 기준. KST.

08:00 KST · cron 1번째 trigger
윈도우 (08:38 ~ 08:42, ±2분) 밖 → skip. runs 추가 X.
08:05 ~ 08:35 KST · cron 2~8번째
매 5분 trigger 지만 모두 윈도우 밖 (≥3분 차이) → skip.
08:40 KST · cron 9번째 ⭐
정확 매칭 → 파이프라인 진행 → 메일 발송. runs 에 success row 추가.
08:45 ~ 09:55 KST · cron 10~22번째
윈도우 밖 또는 has_success_today=True → skip. 중복 발송 0.
다음날 00:00 KST
새 KST 일자 시작. has_success_today 리셋. 08:40 에 다시 발송.

09트러블슈팅

자주 발생하는 케이스 5개.

Q1. cron 12회 모두 skip — 한 번도 발송 안 됨

가능한 원인 3가지:

  • send_hour:send_minute 가 윈도우 밖 — admin Settings 에서 KST 08:00~09:55 범위 안으로 변경 (e.g. 8:40).
  • scrape_enabled=OFF — Overview 탭의 토글 ON 으로 변경.
  • subscribers 비어 있음 — Subscribers 탭에서 최소 1명 추가.

Q2. 같은 날 두 번 발송됨

이론상 불가능 (has_success_today 차단). 만약 발생하면:

  • runs 테이블에 finished_at 이 null 인 zombie row 가 있는지 SQL Editor 확인.
  • 강제발송 (admin 버튼) 은 의도적으로 중복 허용 — 사람 트리거 동작.

Q3. GitHub Actions cron 자체가 안 깨어남

GitHub free tier 의 known issue — repo 가 60일 inactive 면 cron 자동 disable.

  • 해결: Actions 탭 → Daily AI News Digest → Enable workflow 클릭.
  • 예방: 가끔 commit 1개 push 또는 수동 trigger 1회.

Q4. admin Settings 에 send_hour 입력 칸이 안 보임

마이그레이션 0004 미적용 또는 admin 서버 재시작 안 함.

  • Supabase SQL Editor 에서 §4 의 SQL 재실행 (멱등 — 두 번 OK).
  • 로컬 admin 인 경우 make admin 재시작.

Q5. GitHub Actions 분 cap 걱정

private repo free tier 월 2,000분. 현재 설계:

  • 일 22회 trigger × ~1분/회 + 발송 ~2분/일 = 월 ~720분 (36%). 안전 마진 충분.
  • public repo 면 무제한.

10최종 체크리스트

대표님 본인 점검 (운영 시작 전)

  • GitHub repo Settings → Secrets and variables → Actions 에 8개 secret 보임
  • Supabase SQL Editor 에서 마이그레이션 0004 실행 + Table Editor 에서 send_hour/send_minute 컬럼 확인
  • GitHub Actions 탭 → Daily AI News Digest → Run workflow 수동 1회 성공 (또는 outside window skip)
  • 본인 Gmail 받은편지함에 한국어 요약 메일 도착 (수동 trigger 또는 admin 강제발송)
  • admin Settings 탭 → 발송 시각 입력 칸 2개 보이고 저장 동작
  • admin Overview 탭 첫 행 "발송 시각: 08:40 KST" 표시
  • 다음날 자동 발송 1회 확인 — admin History 탭 또는 메일
✅ 완료 이 가이드대로 다 하셨으면 매일 아침 8:40 KST (또는 admin 에서 정한 시각) 에 자동 발송이 가동됩니다. 시간 바꾸고 싶을 땐 admin Settings 에서 입력 + 저장만 하시면 됩니다.
